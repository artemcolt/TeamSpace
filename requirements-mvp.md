# Team Space MVP Requirements

Дата: 2026-05-27

## 1. Назначение продукта

Team Space MVP - локальное desktop-приложение для личной автоматизации работы с рабочими сообщениями и задачами.

Основной сценарий:

> Пользователь выбирает одно или несколько сообщений из Telegram, приложение формирует редактируемый черновик задачи, пользователь проверяет данные и создает задачу в Redmine.

Продукт не должен требовать изменения процессов всей команды. На первом этапе он решает личную боль пользователя: быстро переносить поручения и договоренности из Telegram в Redmine, не теряя контекст.

## 2. Проблема

В рабочем процессе задачи и поручения часто прилетают в Telegram: в личку, рабочие чаты или обсуждения. Чтобы превратить сообщение в задачу Redmine, нужно вручную:

- найти сообщение;
- скопировать текст;
- открыть Redmine;
- выбрать проект;
- заполнить тему, описание, приоритет и другие поля;
- сохранить задачу;
- помнить, что сообщение уже обработано.

Из-за ручной работы часть поручений теряется, задачи создаются поздно или без достаточного контекста.

## 3. Цель MVP

Проверить гипотезу:

> Если упростить создание Redmine-задач из Telegram-сообщений до одного короткого сценария с подтверждением, пользователь будет чаще и быстрее фиксировать поручения в таск-трекере.

Успех MVP:

- пользователь создает реальные Redmine-задачи через приложение;
- создание задачи занимает меньше времени, чем вручную;
- пользователь возвращается к приложению без внешнего напоминания;
- хотя бы один коллега после демонстрации хочет попробовать инструмент на своих задачах.

## 4. Целевая аудитория MVP

Первичная аудитория:

- один пользователь-разработчик/технический специалист;
- коллега из той же рабочей среды, которому также прилетают задачи в Telegram.

Не целевая аудитория MVP:

- вся компания;
- руководители как администраторы процесса;
- команды, которые требуют централизованного внедрения;
- пользователи без Redmine.

## 5. Ключевой пользовательский сценарий

1. Пользователь запускает desktop-приложение.
2. Пользователь подключает Telegram-аккаунт.
3. Пользователь подключает Redmine через URL и API key.
4. При подключении Telegram пользователь выбирает папку Telegram с рабочими чатами.
5. Приложение показывает последние сообщения из выбранных чатов.
6. Пользователь выбирает одно или несколько сообщений.
7. Пользователь нажимает "Создать задачу".
8. Приложение формирует черновик задачи.
9. Пользователь редактирует тему, описание, проект, приоритет и другие поля.
10. Пользователь нажимает "Создать в Redmine".
11. Приложение создает задачу через Redmine API.
12. Приложение сохраняет локальную связь: выбранные сообщения -> созданная Redmine-задача.
13. Приложение помечает сообщения как обработанные.

## 6. Функциональные требования

### 6.1. Telegram integration

Приложение должно:

- подключаться к Telegram как пользовательский клиент, а не как bot, если требуется доступ к личным сообщениям и обычным рабочим чатам;
- поддерживать авторизацию по номеру телефона, коду подтверждения и 2FA-паролю, если он включен;
- получать список доступных диалогов/чатов пользователя;
- получать и отображать папки Telegram, если Telegram API/выбранная библиотека это поддерживает;
- при первичной настройке предлагать выбрать папку с рабочими чатами;
- после выбора папки использовать чаты из этой папки как рабочий набор по умолчанию;
- позволять вручную включить/исключить отдельные рабочие чаты, если папки недоступны или состав папки нужно уточнить;
- загружать последние сообщения из выбранных чатов;
- отображать текст сообщения, автора, дату, название чата;
- позволять выбрать одно или несколько сообщений для создания задачи;
- хранить Telegram session локально и безопасно.

MVP не обязан:

- читать все сообщения в фоне;
- автоматически определять поручения;
- работать с голосовыми сообщениями;
- распознавать скриншоты;
- отвечать в Telegram;
- создавать Telegram-ссылки на сообщения.

### 6.2. Redmine integration

Приложение должно:

- подключаться к Redmine по base URL и API key;
- по умолчанию предлагать Redmine URL: `https://redmine.example.com/`;
- объяснять пользователю, где взять API key: `https://redmine.example.com/my/account`;
- проверять валидность подключения;
- получать список проектов, доступных пользователю;
- получать список trackers, priorities и users, если это доступно через API;
- создавать issue в выбранном проекте;
- сохранять ID и URL созданной задачи локально;
- показывать пользователю результат создания задачи.

В MVP допустимо:

- задать проект по умолчанию;
- задать tracker по умолчанию;
- задать priority по умолчанию;
- не поддерживать все custom fields, если они не обязательны для создания задачи.

Если в конкретном Redmine есть обязательные custom fields, приложение должно явно показать ошибку и не терять черновик.

Для текущей рабочей среды:

- Redmine base URL: `https://redmine.example.com/`;
- страница API key пользователя: `https://redmine.example.com/my/account`;
- API key находится в настройках аккаунта в поле "API-ключ";
- экран настройки Redmine должен содержать подсказку по этой ссылке и по возможности иллюстрацию/текстовое описание, где находится ключ.

### 6.3. Черновик задачи

Приложение должно формировать редактируемый черновик перед созданием задачи.

Минимальные поля черновика:

- тема;
- описание;
- проект;
- tracker;
- priority;
- assignee, если доступно;
- selected source messages.

Описание задачи должно включать исходный контекст без Telegram-ссылки:

```text
Источник: Telegram
Чат: <название чата>
Автор: <имя автора>
Дата: <дата и время сообщения>

Сообщение:
<текст сообщения>
```

Если выбрано несколько сообщений, они должны добавляться в описание в хронологическом порядке.

### 6.4. Локальное состояние

Приложение должно хранить локально:

- выбранные рабочие чаты;
- последние загруженные сообщения;
- статус сообщения: new, ignored, drafted, created;
- созданные черновики;
- связь message ids -> Redmine issue id;
- настройки Redmine без секретов;
- пользовательские defaults для проекта, tracker, priority.

Секреты должны храниться отдельно:

- Telegram session;
- Redmine API key;
- возможные AI provider tokens.

Секреты нельзя хранить в обычном JSON/plain text файле.

### 6.5. AI draft generation

AI не является обязательным для MVP.

MVP должен быть полезен без AI:

- копирует выбранные сообщения в описание;
- предлагает тему простым правилом или оставляет поле для ручного ввода;
- заполняет defaults.

Опциональный AI-режим:

- local Ollama/LM Studio provider;
- без отправки сообщений во внешнее облако;
- AI предлагает тему, краткое описание, список действий и открытые вопросы;
- пользователь обязан подтвердить черновик перед созданием задачи.

В MVP запрещено:

- автоматически создавать задачи без подтверждения пользователя;
- отправлять текст Telegram-сообщений в облачную модель без явного включения и предупреждения;
- скрывать от пользователя, какой AI provider используется.

## 7. Нефункциональные требования

### 7.1. Privacy

Все рабочие данные должны обрабатываться локально на компьютере пользователя.

По умолчанию приложение не должно отправлять содержимое Telegram-сообщений во внешние сервисы, кроме Redmine при создании задачи.

### 7.2. Security

Приложение должно:

- хранить секреты через системное защищенное хранилище, если доступно;
- не логировать Telegram session, Redmine API key и коды авторизации;
- не показывать секреты в UI после сохранения;
- иметь явную кнопку отключения Telegram и Redmine;
- позволять удалить локальные данные приложения.

### 7.3. Reliability

Приложение должно:

- не терять черновик при ошибке создания задачи;
- показывать понятную ошибку при недоступности Telegram или Redmine;
- не создавать дубли случайным двойным нажатием;
- помечать сообщение как created только после успешного ответа Redmine.

### 7.4. Performance

MVP должен комфортно работать на обычном рабочем ноутбуке.

Ориентиры:

- старт приложения до 5 секунд;
- загрузка последних сообщений выбранного чата до 3 секунд при нормальной сети;
- создание Redmine-задачи до 5 секунд при нормальной сети.

## 8. UX требования

### 8.1. First-run onboarding

При первом запуске приложение должно предложить пользователю настроить интеграции.

Onboarding должен быть пошаговым:

1. Экран приветствия с кратким объяснением сценария: Telegram messages -> Redmine issue draft -> create issue.
2. Экран настройки Telegram.
3. Экран выбора рабочей Telegram-папки или рабочих чатов.
4. Экран настройки Redmine.
5. Экран выбора default project/tracker/priority.
6. Финальный экран проверки настроек.

Пользователь должен иметь возможность пропустить настройку Redmine или Telegram, но основной сценарий создания задач доступен только после подключения обеих интеграций.

### 8.2. Telegram configuration screen

Экран настройки Telegram должен содержать:

- ввод номера телефона;
- ввод кода подтверждения;
- ввод 2FA-пароля, если Telegram его запросит;
- список Telegram-папок после успешной авторизации;
- выбор папки с рабочими чатами;
- fallback-режим ручного выбора чатов, если папки недоступны;
- кнопку проверки подключения;
- кнопку отключения Telegram и удаления локальной session.

Выбор папки рабочих чатов является частью первичной настройки Telegram.

### 8.3. Redmine configuration screen

Экран настройки Redmine должен содержать:

- поле Redmine URL;
- предзаполненное значение `https://redmine.example.com/`;
- поле API key;
- ссылку-подсказку `https://redmine.example.com/my/account`;
- текстовую подсказку: "Откройте страницу аккаунта Redmine и скопируйте значение из поля API-ключ";
- кнопку проверки подключения;
- статус подключения;
- кнопку сохранения настроек;
- кнопку отключения Redmine и удаления API key из локального хранилища;
- после успешного подключения - выбор проекта, tracker и priority по умолчанию.

При наличии скриншота-подсказки из Redmine account settings его можно использовать в интерфейсе onboarding или документации, чтобы показать пользователю, где находится поле "API-ключ".

### 8.4. Main screen

Основной экран должен содержать:

- список выбранных Telegram-чатов;
- список сообщений выбранного чата;
- фильтр по статусам: new, ignored, created;
- кнопку создания задачи из выбранных сообщений;
- индикатор уже обработанных сообщений.

### 8.5. Issue draft screen

Экран черновика должен содержать:

- исходные сообщения;
- редактируемые поля задачи;
- выбор проекта/tracker/priority;
- кнопку "Создать в Redmine";
- кнопку "Сохранить черновик";
- кнопку "Отмена".

После создания задачи приложение должно:

- показать ID задачи;
- дать возможность открыть задачу в браузере;
- пометить исходные сообщения как created.

## 9. Ограничения MVP

В MVP не делаем:

- корпоративное внедрение;
- замену Outline/Redmine/GitLab;
- полноценную wiki;
- полноценный таск-трекер;
- командную синхронизацию между пользователями;
- web-версию;
- мобильную версию;
- автоматический анализ всех сообщений;
- автосоздание задач без ручного подтверждения;
- обязательный cloud AI;
- Telegram bot mode как основной сценарий для личных сообщений.

## 10. Архитектурные предположения

Текущий проект: Electron + React + TypeScript.

Предпочтительная структура:

- Electron main process: Telegram client, Redmine API, локальная база, keychain;
- Electron renderer: UI;
- preload: безопасный IPC bridge;
- SQLite: локальная база приложения;
- OS keychain/keytar-compatible storage: секреты;
- AI provider interface: off/local/cloud в будущем.

Все интеграции с внешними системами должны идти через main process, а не напрямую из renderer.

## 12. MVP acceptance criteria

MVP можно считать готовым для личного тестирования, если:

- пользователь может подключить Telegram;
- пользователь может выбрать Telegram-папку с рабочими чатами или вручную выбрать рабочие чаты;
- приложение показывает последние текстовые сообщения;
- пользователь может выбрать сообщение;
- пользователь может открыть экран настройки Redmine;
- экран настройки Redmine предлагает `https://redmine.example.com/` как base URL;
- экран настройки Redmine содержит подсказку, что API key находится на `https://redmine.example.com/my/account`;
- пользователь может подключить Redmine через API key;
- приложение получает список проектов или позволяет задать проект вручную;
- пользователь может открыть черновик задачи;
- черновик содержит текст сообщения и контекст: чат, автор, дата;
- пользователь может отредактировать тему и описание;
- приложение создает задачу в Redmine;
- приложение сохраняет локальную связь сообщения и Redmine issue;
- повторное нажатие не создает дубль без явного действия пользователя;
- секреты не лежат в plain text файлах.

## 13. Метрики проверки гипотезы

В течение первой недели личного использования замерить:

- сколько задач создано через приложение;
- среднее время создания задачи через приложение;
- среднее время ручного создания аналогичной задачи;
- сколько сообщений помечено ignored;
- сколько черновиков не было создано в Redmine;
- сколько раз пользователь вернулся в приложение без напоминания;
- захотел ли коллега попробовать инструмент после демонстрации.

Минимальный положительный сигнал:

- приложение используется минимум 3 рабочих дня подряд;
- создано минимум 5 реальных задач;
- пользователь субъективно считает сценарий быстрее ручного копирования;
- коллега готов протестировать на своих сообщениях.

## 14. Открытые вопросы

- Какая библиотека Telegram client API будет использоваться в Electron/Node окружении?
- Какие обязательные поля настроены в рабочем Redmine?
- Нужно ли приложение только для macOS или сразу для Windows/Linux?
- Можно ли использовать локальный keychain на всех целевых ОС?
- Нужен ли импорт сообщений из Telegram Desktop, если Telegram API окажется неудобным?
- Нужна ли поддержка нескольких Redmine instance?
- Будет ли AI в первой версии или только шаблонный черновик?
- Какой локальный AI provider использовать первым: Ollama или LM Studio?

## 15. Следующая итерация после MVP

Если MVP подтвердит пользу:

- создание задачи из нескольких сообщений;
- группировка сообщений по теме;
- локальный AI для извлечения action items;
- daily inbox с необработанными поручениями;
- напоминания о сообщениях, которые пользователь пометил later;
- генерация комментария в Redmine из последующих Telegram-сообщений;
- связь Redmine issue с GitLab branch/MR;
- экспорт локальной истории в markdown.
